# ARPG Demo 项目架构重构指南
# 项目所有脚本在Assets/Scripts下

## 文档概述

本文档描述了如何将现有的 ARPG Demo 项目从传统的单例管理器架构重构为基于 MVC/MVCS 分层架构的设计。这是一个渐进式的重构计划，旨在提高代码的可维护性、可测试性和可扩展性，同时保持项目的核心设计理念和已有的优秀设计模式。

## 原项目架构分析

### 当前架构特点

你的项目目前采用了一种混合的架构模式，包含以下核心特征。项目中的各个功能模块通过单例管理器来组织，比如任务系统有 QuestManager、UI 系统有 UIManager、音效系统有 AudioManager 等。这些管理器都是全局可访问的单例，负责各自领域的功能实现。各个管理器之间通过一个中心化的事件系统进行通信，实现了模块间的解耦。这种设计让模块之间不需要直接引用彼此，而是通过发布订阅模式来协作。

玩家系统采用了一种类似中介者模式的设计，PlayerManager 作为中心协调器，管理着玩家的多个子系统，包括攻击、动画、库存、移动、输入等功能。这些子系统不直接相互调用，而是通过 PlayerManager 来协调它们之间的交互。这种设计在一定程度上控制了子系统之间的复杂度。

武器系统使用了基于 ScriptableObject 的策略模式。每个武器可以配置不同按键对应的动作行为，这些动作被封装成继承自 ScriptableObject 的策略类。这种设计提供了极高的灵活性，可以在编辑器中可视化地配置不同武器的行为，而不需要修改代码。

数据配置同样采用 ScriptableObject 来管理，包括所有物品数据、任务数据、对话数据等。这些配置数据在编辑器中可视化编辑，运行时通过 Resources 或 AssetBundle 加载。

多存档功能已经实现，可以支持玩家创建和切换不同的游戏存档。

### 现有架构的优点

这个架构在实际项目中已经证明了它的可行性。事件驱动的设计让模块之间保持了松耦合，你可以独立开发和测试不同的功能模块。基于 ScriptableObject 的配置系统提供了强大的数据驱动能力，策划和美术可以在不修改代码的情况下调整游戏内容。PlayerManager 的中介者模式有效地管理了玩家相关功能的复杂性，避免了子系统之间的混乱依赖。

### 现有架构面临的挑战

然而随着项目规模的增长，一些问题开始显现。单例模式虽然方便，但带来了全局状态的问题，任何地方都可以访问和修改这些管理器，导致依赖关系不够清晰。测试变得困难，因为很难模拟或替换单例管理器来进行单元测试。各个管理器的职责边界有时不够清晰，一个管理器可能既负责数据管理，又负责业务逻辑，还负责表现层的控制，职责混在一起难以维护。

资源加载方式混用了 Resources 和 AssetBundle，缺乏统一的资源管理策略，导致资源加载逻辑分散在各处，难以统一管理和优化。当你想要添加热更新功能或优化资源加载性能时，会发现需要修改很多地方的代码。

虽然事件系统实现了模块解耦，但缺少更结构化的架构约束。什么时候应该用事件，什么时候可以直接调用，没有明确的指导原则，导致代码风格不够统一。

## 目标架构概述

### 新框架的核心理念

我们将采用的新框架是一个基于分层架构的设计，它将应用程序分为四个主要层次，每一层都有明确的职责边界和访问权限。这个框架的核心思想是通过清晰的分层和接口约束来组织代码，让每个部分的职责变得明确，同时提供足够的灵活性来实现复杂的业务逻辑。

Model 层是数据和业务规则的持有者。它负责管理应用程序的核心数据状态，比如玩家的属性、背包物品、任务进度等。Model 层被设计为相对封闭的，它只能获取 Utility 工具类和发送事件。这种限制确保了数据层的纯净性，数据的变化通过事件通知出去，而不是直接调用其他层的方法。

System 层充当业务逻辑的处理中心。它可以访问 Model 获取和修改数据，调用 Utility 使用工具功能，发送事件通知其他部分，还能获取其他 System 来实现系统间的协作。System 层负责处理复杂的业务流程，协调多个模块来完成特定的功能。

Controller 层是表现层的控制器，通常对应 Unity 中的 MonoBehaviour 组件。它能发送命令来执行操作，发送查询来获取数据，获取 System 和 Model 来访问业务逻辑和数据，注册事件来响应状态变化。Controller 是用户交互的入口，当玩家点击按钮或触发某个行为时，Controller 将这些操作转化为命令发送给架构执行。

Utility 层提供纯粹的工具功能，比如存储服务、配置读取、数学计算、字符串处理等。它不依赖任何其他层，是最底层的功能提供者。Utility 应该是无状态的或状态完全自包含的，可以在任何项目中复用。

### 架构的核心机制

除了分层结构，框架还提供了三种重要的交互机制。

命令模式用于封装完整的业务操作。当用户触发一个动作时，比如购买物品、使用技能、升级角色，这些操作都应该封装成命令。命令可以访问 Model、System、Utility，可以发送其他命令和查询，可以发送事件。命令让业务流程变得可追踪、可测试、可扩展。

查询模式用于封装复杂的数据获取逻辑。当需要计算战斗力评分、检查任务完成条件、获取排行榜数据等需要组合多个数据源的信息时，使用查询可以让这些计算逻辑独立出来，便于复用和测试。

事件模式用于状态变化的通知。当数据发生变化时，比如玩家升级、物品被添加、任务完成，相关的模块需要知道这个变化。事件是解耦的关键机制，发送者不需要知道谁在监听，监听者也不需要知道是谁发送的。

框架还提供了 BindableProperty 来实现数据绑定。当 Model 中的数据使用 BindableProperty 包装后，UI 可以监听这些属性的变化并自动更新显示，无需手动编写刷新逻辑。

### 重构的核心目标

通过这次重构，我们希望达到以下几个目标。首先是清晰的职责分离，每个类、每个层次都有明确的职责，不会出现一个类既管数据又管逻辑还管表现的情况。其次是改善的可测试性，通过依赖注入和接口抽象，可以轻松地为各个组件编写单元测试。第三是更好的可维护性，当需要修改某个功能时，能够快速定位到相关代码，修改的影响范围是可预测的。第四是统一的资源管理，引入 YooAsset 作为统一的资源加载方案，为将来的热更新和资源优化打好基础。最后是保留优秀的设计，你原项目中基于 ScriptableObject 的策略模式和数据配置系统非常优秀，我们会保留这些设计并让它们与新架构和谐共存。

## 核心概念详解

### Model 层的设计原则

Model 层在新架构中扮演着数据权威来源的角色。理解 Model 的本质非常重要，它不是用来表示单个游戏对象的数据容器，而是用来管理整个应用程序中某一类业务数据的管理者。

对于全局唯一的数据，比如玩家数据、游戏配置、全局设置等，我们直接用 Model 来存储。PlayerModel 存储唯一玩家的等级、生命值、体力等信息，这是整个游戏中的单一数据源。任何需要读取或修改玩家数据的地方都应该通过 PlayerModel 来进行。

对于存在多个实例的实体数据，比如敌人、NPC、掉落物等，我们不会为每个实例创建一个 Model。相反，我们创建一个管理器 Model 来管理这些实例的集合。比如 EnemyManagerModel 内部维护一个字典，存储所有活跃敌人的数据。每个敌人的数据用一个普通的数据类来表示，而不是继承 Model。这样的设计让数据的组织层次清晰，既保持了集中管理的优点，又避免了为每个实例创建 Model 带来的复杂性。

Model 可以包含与数据直接相关的业务规则和验证逻辑。比如 InventoryModel 在添加物品时检查背包是否已满，检查物品是否可堆叠，这些都是维护数据完整性的逻辑，应该放在 Model 中。但是 Model 不应该处理跨系统的复杂业务流程，那些应该由 System 或 Command 来协调。

Model 使用 BindableProperty 来包装需要被观察的数据。当这些数据发生变化时，BindableProperty 会自动通知所有注册的监听者。这让 UI 能够实现自动更新，无需手动编写大量的刷新代码。

Model 可以发送事件来通知重要的状态变化。比如当玩家生命值降为零时，PlayerModel 发送 PlayerDeathEvent。但 Model 不能获取其他 Model 或 System，这保证了 Model 层的独立性和可测试性。

### System 层的设计原则

System 层是业务逻辑的处理中心，它负责协调多个模块完成复杂的功能。理解 System 和 Model 的区别是掌握这个架构的关键。

System 处理的是需要协调多个模块的业务流程。比如当玩家使用一个治疗药水时，这个流程涉及到从背包移除物品、恢复玩家生命值、播放使用特效、播放音效、可能还要触发成就检查。这些跨模块的协调逻辑应该由 System 来处理。

System 可以有状态，可以维护一些业务运行时需要的数据。比如 CombatSystem 可能需要记录当前的连击数、技能冷却状态等。但这些状态通常是临时的、计算性的，而不是需要持久化的核心数据。核心数据应该在 Model 中。

System 可以监听事件来响应状态变化。比如 QuestSystem 监听敌人死亡事件来更新任务进度，监听物品获得事件来检查收集类任务。通过事件驱动的方式，System 之间可以松耦合地协作。

System 可以调用其他 System 的方法来实现功能组合。但要注意避免循环依赖，如果两个 System 需要频繁相互调用，可能说明职责划分不够清晰，需要重新思考设计。

### Utility 层的设计原则

Utility 提供纯粹的工具性功能，这些功能应该是业务无关的、可复用的。判断某个功能是否应该是 Utility 可以问几个问题。这个功能需要维护状态吗？如果需要记住之前的操作、需要管理一个集合、需要追踪某些数据的变化，那它应该是 System。这个功能需要与架构的其他部分交互吗？如果需要获取 Model 的数据、需要调用其他 System、需要发送或监听事件，那它应该是 System。这个功能是否具有通用性？如果这个功能完全独立，可以拿到任何项目中使用，不包含特定的业务逻辑，那它可能是 Utility。

典型的 Utility 包括配置读取工具，它负责加载和提供 ScriptableObject 配置数据。存储工具负责文件的读写、JSON 序列化等。数学工具提供随机数生成、向量计算、插值等功能。字符串工具提供格式化、转换等功能。加密工具提供数据的加密和解密。

你的 JsonManager 就是一个典型的 Utility，它提供 JSON 序列化和反序列化的功能，不依赖任何业务逻辑，可以在任何需要 JSON 处理的地方使用。

### Command 和 Query 的使用时机

命令用于封装修改性的操作，也就是会改变系统状态的操作。购买物品、使用技能、升级角色、保存游戏、加载游戏等，这些都应该封装成命令。命令的好处是它封装了完整的业务流程，包括前置检查、状态修改、后续通知等所有步骤。命令是可测试的，你可以创建一个测试环境，执行命令，然后验证状态是否按预期改变。命令也是可扩展的，如果将来需要添加新的逻辑，比如记录操作日志、实现撤销功能，都可以在命令的框架内完成。

查询用于封装复杂的数据获取逻辑，也就是不改变状态只返回结果的操作。计算战斗力、检查任务是否可接取、获取排行榜数据、计算伤害数值等，这些应该封装成查询。查询的好处是它将计算逻辑独立出来，当计算规则改变时只需要修改查询的实现。查询是可复用的，同样的计算逻辑可能在多个地方使用。查询也是可测试的，你可以准备测试数据，执行查询，验证返回结果是否正确。

对于简单的数据读取，比如获取玩家当前生命值，直接访问 Model 的属性即可，不需要封装成查询。对于简单的数据修改，如果只是单纯地改变一个值而没有复杂的业务逻辑，也可以考虑直接在 Model 中提供方法，不一定要封装成命令。架构是为了帮助我们组织代码，而不是增加不必要的复杂性。在实际开发中需要根据具体情况灵活运用。

### Event 的设计原则

事件用于通知状态变化，而不是执行操作。当某个重要的状态发生改变时，发送事件让其他模块知道。但事件本身不应该携带复杂的业务逻辑。

事件应该是语义清晰的。事件的名称应该准确描述发生了什么，比如 PlayerLevelUpEvent、ItemAddedEvent、QuestCompletedEvent。看到事件名就能知道发生了什么事情。

事件应该携带必要的上下文信息。比如 ItemAddedEvent 应该包含物品 ID、数量等信息，让监听者能够做出适当的响应。但不要在事件中携带过多的数据，只包含必要的信息即可。

发送事件的时机很重要。通常在状态已经改变之后发送事件，这样监听者可以安全地读取新的状态。如果在状态改变之前发送，可能导致监听者读取到旧的数据。

事件的监听者应该只做响应式的处理，不应该在事件处理函数中执行复杂的业务逻辑。如果需要执行复杂操作，应该发送命令或调用 System 的方法。

要注意避免事件循环。A 发送事件，B 监听后又发送事件，A 再监听又发送事件，这样会形成死循环。设计时要注意事件的流向，保持单向的依赖关系。

## 模块重构策略

### UI 系统的重构

你原有的 UI 系统通过 UIManager 管理各个面板的生命周期和显示控制。在新架构下，我们将保留 UIManager 的核心功能，但将它转变为一个更专注的 UISystem。

UISystem 主要负责面板的生命周期管理，包括打开面板、关闭面板、管理面板栈来实现返回上一个界面的功能、处理面板的层级排序。UISystem 不再负责面板内部的数据处理和业务逻辑，那些应该由各个面板对应的 Controller 来处理。

每个 UI 面板成为一个独立的 Controller。比如你有一个背包面板，就创建一个 InventoryPanelController。这个 Controller 持有面板上所有 UI 组件的引用，负责监听数据变化并更新显示，处理用户在面板上的交互操作。

Controller 通过 GetModel 来获取需要显示的数据。比如 InventoryPanelController 获取 InventoryModel，监听背包物品的变化。当物品列表改变时，Controller 自动刷新显示。这种数据绑定的方式让 UI 的更新变得简单和可靠。

当用户点击 UI 上的按钮时，Controller 发送命令来执行操作。比如用户点击使用物品按钮，Controller 发送 UseItemCommand。命令内部会处理所有的业务逻辑，包括检查是否可以使用、扣除物品、应用效果等。Controller 本身不需要知道这些细节，它只负责将用户的意图转化为命令。

你原来使用外观类来让 UI 访问玩家数据的设计可以去掉了。在新架构下，UI Controller 直接通过 GetModel 来访问需要的数据，不需要中间的外观类。这样数据的流向更加直接和清晰。

### 任务系统的重构

任务系统是一个典型的需要同时管理配置数据和运行时状态的模块。我们会将它拆分成几个部分。

QuestConfigUtility 负责管理所有任务的静态配置数据。这些配置来自 ScriptableObject，包含任务的描述、目标、奖励等信息。Utility 在初始化时加载所有任务配置，然后提供各种查询接口，比如根据 ID 获取任务配置、获取某个类型的所有任务、检查任务的前置条件等。

QuestModel 负责管理玩家当前的任务状态。它存储玩家接受了哪些任务、每个任务的进度如何、完成了哪些任务。这些都是运行时的状态数据，需要持久化到存档中。QuestModel 提供方法来接受任务、更新任务进度、完成任务等。当任务状态改变时，Model 发送相应的事件。

QuestSystem 负责处理任务相关的业务逻辑。比如当玩家接受一个任务时，QuestSystem 检查是否满足接受条件、是否达到任务数量上限等。当任务完成时，QuestSystem 负责发放奖励，包括给予经验值、金币、物品等。QuestSystem 还会监听各种游戏事件来更新任务进度，比如监听敌人死亡事件来更新击杀任务、监听物品获得事件来更新收集任务。

UI 层面，QuestListController 显示任务列表，监听 QuestModel 的变化来自动更新显示。QuestDetailController 显示任务详情。当玩家点击接受任务按钮时，Controller 发送 AcceptQuestCommand。

这种分层设计让任务系统的各个部分职责清晰。配置数据的读取、运行时状态的管理、业务逻辑的处理、UI 的显示，每一部分都在自己的层次中，互不干扰又能很好地协作。

### 库存系统的重构

库存系统同样需要处理配置数据和运行时状态。ItemConfigUtility 管理所有物品的配置信息，这些配置定义了物品的名称、图标、属性、效果等静态数据。

InventoryModel 管理玩家背包的运行时状态。它存储背包中有哪些物品、每个物品的数量、物品在哪个槽位等。InventoryModel 提供方法来添加物品、移除物品、使用物品、整理背包等。当背包内容改变时，Model 通过 BindableProperty 通知变化。

InventorySystem 处理与背包相关的复杂业务逻辑。比如当玩家使用一个消耗品时，System 需要协调多个模块，从背包移除物品、应用物品效果、播放特效和音效、可能还要触发成就检查。这些跨模块的协调工作由 System 来完成。

UI 方面，InventoryPanelController 显示背包界面，监听 InventoryModel 的变化来更新显示。当用户拖拽物品、点击使用按钮时，Controller 发送相应的命令。

装备系统可以作为库存系统的扩展。EquipmentModel 存储当前装备的物品，提供装备和卸下的方法。EquipmentSystem 处理装备时的属性加成计算、装备条件检查等逻辑。

### 玩家系统的重构

玩家系统是你原项目中最复杂的部分，PlayerManager 作为中介者协调着攻击、动画、库存、移动、输入等多个子系统。在新架构下，我们会保持这种组件化的设计思想，但让各个部分的职责更加清晰。

PlayerModel 存储玩家的核心数据，包括等级、生命值、魔法值、体力、属性点等。这些数据使用 BindableProperty 包装，任何变化都会自动通知监听者。PlayerModel 还存储一些状态标志，比如是否在冲刺、是否双手持武器、当前使用哪只手等。这些状态既是业务数据，也会影响表现层的行为。

PlayerSystem 可以作为一个协调者，处理一些需要跨多个子系统的逻辑。但它不再像原来的 PlayerManager 那样中心化地管理一切，而是更专注于玩家特有的业务逻辑，比如升级时的奖励发放、属性点的分配、状态效果的管理等。

表现层仍然保持组件化的设计。CharacterController 是玩家 GameObject 上的根组件，它组合了 CharacterMovementComponent、CharacterCombatComponent、CharacterAnimationComponent 等功能组件。这些组件专注于表现层的逻辑，比如播放动画、处理移动、检测碰撞等。

CharacterController 实现 IController 接口，这让它能够访问架构。当需要读取玩家数据时，组件通过 Controller 来获取 Model。当需要执行业务操作时，组件通过 Controller 来发送命令。组件也可以监听事件来响应状态变化。

CharacterController 需要从 Model 同步一些状态到表现层。比如当 PlayerModel 中的 IsSprinting 改变时，CharacterController 更新自己的 isSprinting 字段，然后各个子组件可以根据这个状态来调整行为。这种数据冗余是必要的，因为表现层和数据层应该保持一定的隔离。

移动系统可以继续使用你原有的组件化设计。CharacterMovementComponent 负责移动的表现，包括设置速度、旋转方向等。移动的输入可以来自玩家输入，也可以来自 AI 系统。组件本身不关心输入从哪里来，它只负责执行移动。

战斗系统同样保持组件化。CharacterCombatComponent 负责攻击的表现，包括检测攻击范围内的目标、触发命中检测等。当检测到命中时，组件通知业务层来计算和应用伤害。业务层的伤害计算、属性修改等逻辑由 CombatSystem 或命令来处理。

动画系统的 CharacterAnimationComponent 负责播放各种动画。它提供统一的接口来播放移动动画、攻击动画、受击动画等。具体播放哪个动画、动画的参数如何设置，由调用者决定。

### 输入系统的重构

输入系统是连接用户操作和游戏逻辑的桥梁。你使用的是 Unity 的新输入系统 InputSystem，这是一个很好的选择。在新架构下，输入处理会变得非常清晰。

PlayerInputController 是一个 Controller 组件，负责监听所有的输入事件。它使用 Input Actions 来定义各种输入，比如移动、跳跃、攻击、使用物品等。当输入事件触发时，Controller 将输入转化为命令发送给架构。

比如当玩家按下鼠标左键时，触发攻击输入事件。PlayerInputController 的事件处理函数被调用，它发送一个 PerformWeaponActionCommand，这个命令会获取玩家当前装备的武器、获取武器对应按键的动作配置、检查是否可以执行动作比如体力是否足够、执行体力消耗等业务逻辑、最后调用动作策略来执行表现层的逻辑。

这种设计的好处是输入处理和业务逻辑完全分离。InputController 只负责监听输入并转化为命令，它不需要知道业务逻辑的细节。命令封装了完整的业务流程，可以被测试、可以被复用。如果将来要支持其他输入方式，比如手柄或触摸屏，只需要创建新的 InputController 来监听不同的输入源，发送同样的命令即可。

移动输入可以直接调用 CharacterMovementComponent 的方法。当检测到移动输入时，InputController 获取移动向量，然后调用移动组件的 Move 方法。移动这种每帧都在发生的操作不需要通过命令系统，直接调用组件方法更高效。

### 基于 ScriptableObject 的动作策略处理

你的动作策略系统设计得很好，我们会保留这个设计并让它与新架构融合。关键是理解策略模式和命令模式的不同作用。

动作策略专注于定义动作的执行方式，也就是表现层的逻辑。LightAttackAction_SO 定义了轻攻击应该播放什么动画、如何处理连击、冲刺攻击有什么不同等。这些是表现层的行为定义，可以通过 ScriptableObject 在编辑器中灵活配置。

命令负责将动作整合到业务流程中。PerformWeaponActionCommand 负责业务层面的检查和处理，包括检查体力是否足够、检查当前状态是否允许攻击、消耗体力、发送事件通知等。然后命令调用动作策略来执行表现层的逻辑。

为了让动作策略能够访问它需要的信息，我们建议使用最小化上下文的设计。上下文对象只包含 CharacterManager 的引用，动作策略通过 CharacterManager 来访问需要的状态和组件。CharacterManager 负责从 Model 同步必要的状态，比如 isSprinting、isTwoHandingWeapon 等。这样动作策略的代码改动最小，基本上可以继续使用你原有的逻辑。

武器配置继续使用 ScriptableObject，每个武器定义不同按键对应的动作。这种数据驱动的设计非常灵活，在新架构下依然适用。唯一的变化是调用动作的方式从直接调用变成了通过命令来调用，但这个变化让业务逻辑和表现逻辑得到了更好的分离。

### 音效和特效系统的重构

音效系统和特效系统是典型的 System 层组件。AudioSystem 负责管理音频的播放，包括背景音乐、音效、语音等。它可以提供方法来播放音效、停止音效、调整音量等。其他模块需要播放音效时，可以直接调用 AudioSystem 的方法，或者发送事件让 AudioSystem 监听。

VFXSystem 负责管理特效的生成和管理。当需要播放特效时，比如攻击命中、技能施放、升级特效等，可以调用 VFXSystem 的方法或发送特效事件。VFXSystem 可以使用对象池来管理特效对象，避免频繁的实例化和销毁。

这两个系统都相对独立，它们提供的是服务性的功能。其他模块在需要的时候调用它们，而不是持有它们的引用。这种设计让音效和特效系统可以独立开发和测试，也方便将来替换实现方式。

### 存档系统的重构

存档系统涉及到数据的序列化和持久化，需要协调多个 Model 来收集需要保存的数据。

SaveUtility 提供基础的文件操作能力，包括读取文件、写入文件、检查文件是否存在等。这是纯工具性的功能，不涉及具体的业务逻辑。

JsonUtility 提供 JSON 的序列化和反序列化功能。你原项目中的 JsonManager 可以直接改造成 JsonUtility 使用。

SaveGameCommand 负责执行保存操作。它从各个 Model 收集需要保存的数据，包括 PlayerModel 的玩家数据、InventoryModel 的背包数据、QuestModel 的任务数据等。然后使用 JsonUtility 将数据序列化成 JSON 字符串，使用 SaveUtility 写入到文件中。保存成功后发送 GameSavedEvent。

LoadGameCommand 负责执行加载操作。它使用 SaveUtility 读取文件内容，使用 JsonUtility 反序列化成数据对象，然后将数据分发到各个 Model 中恢复状态。加载成功后发送 GameLoadedEvent。

SaveDataModel 可以用来管理存档元信息，比如存档列表、每个存档的创建时间、游戏进度等。这让玩家可以看到所有存档并选择要加载哪一个。

多存档功能通过不同的文件名来实现，每个存档对应一个文件。SaveGameCommand 和 LoadGameCommand 接收存档槽位名称作为参数，根据槽位名称来读写相应的文件。

### 对象池和计时器系统

对象池应该作为一个 System 来实现。PoolSystem 负责管理多个对象池，每种预制体对应一个池。它提供方法来预热对象池、从池中获取对象、回收对象到池中等。PoolSystem 需要维护池的状态，需要在场景切换时清理对象，这些都是有状态的业务逻辑，所以它应该是 System 而不是 Utility。

计时器同样应该是一个 System。TimerSystem 维护一个定时任务的列表，每帧更新这些任务，检查时间并触发回调。它可以提供方法来创建延迟执行的任务、重复执行的任务、取消任务等。TimerSystem 需要每帧更新，需要维护任务状态，这些特征都表明它应该是 System。

这两个系统都是服务性的系统，其他模块在需要的时候调用它们的方法。比如 VFXSystem 可以使用 PoolSystem 来管理特效对象，CombatSystem 可以使用 TimerSystem 来实现技能冷却。

## 资源管理重构

### 从混合方式到统一管理

你目前的项目混用了 Resources 和 AssetBundle 两种资源加载方式，这确实会带来管理上的混乱。Resources 虽然简单直接，但它会增加包体大小，不支持热更新，而且所有资源都会被打包进去即使不一定会用到。AssetBundle 更灵活，但使用起来比较复杂，需要手动管理依赖关系、加载和卸载等。

统一的资源管理方案可以解决这些问题。我们推荐使用 YooAsset 作为你的资源管理解决方案。YooAsset 是专门为商业项目设计的，它在功能上参考了 Unity 官方的 Addressables，但针对国内项目的实际需求做了很多优化和扩展。

### YooAsset 的核心优势

YooAsset 提供了完整的资源管理工作流程。在编辑器中你可以清晰地配置哪些资源要打包、如何分组、使用什么压缩方式。这种显式的配置让资源管理变得透明，团队成员都能清楚地知道资源是如何组织的。

YooAsset 内置了完善的热更新支持。它提供了版本管理和差异化更新机制，你可以很容易地实现增量更新，只下载有变化的资源文件。这对于需要频繁更新内容的游戏来说非常重要。

资源加密是 YooAsset 的另一个重要特性。你可以对资源包进行加密处理，防止资源被轻易提取。这对于保护美术资源和配置数据很有价值。

YooAsset 的性能经过优化，特别是在资源数量很大的项目中表现出色。它的资源定位算法、依赖管理、内存管理都经过精心设计，能够满足商业项目的性能要求。

YooAsset 提供了详细的中文文档和示例项目，社区也比较活跃。对于中文开发者来说学习成本相对较低，遇到问题也容易找到解决方案。

### 资源管理的架构设计

在新架构中，资源加载应该是透明的，业务逻辑不应该关心资源从哪里来。我们可以创建一个 ResourceSystem 来封装 YooAsset 的使用，提供统一的资源加载接口。

ResourceSystem 负责初始化 YooAsset，根据运行模式选择合适的初始化方式。在编辑器中可以直接从 AssetDatabase 加载，方便开发和调试。在正式包中从本地或远程加载资源包，支持热更新。

ResourceSystem 提供异步加载接口，返回可等待的任务。业务代码可以使用 async/await 来方便地处理异步加载。ResourceSystem 还负责管理资源的生命周期，包括资源的加载、持有、释放等。

对象池系统与资源管理系统配合使用。PoolSystem 在创建对象时通过 ResourceSystem 加载预制体，然后创建实例。对象回收时放回池中，不需要立即释放资源。只有在清空对象池时才释放资源句柄。

配置管理的 Utility 也会使用 ResourceSystem 来加载 ScriptableObject 配置。这样所有的资源加载都通过统一的途径，方便管理和优化。

### 迁移策略

从 Resources 和 AssetBundle 迁移到 YooAsset 应该是一个渐进的过程。首先配置 YooAsset 并让它能够正常工作。在编辑器中创建资源包配置，将你的资源按功能或场景分组。

然后选择一个相对独立的模块开始迁移。比如音效系统，将所有音效资源配置到 YooAsset 中，修改 AudioSystem 的加载代码使用 ResourceSystem。测试确保功能正常后，再迁移下一个模块。

在迁移过程中可以让新旧系统并存。对于已经迁移的资源使用 YooAsset 加载，对于还没迁移的资源继续使用原来的方式。这样可以避免一次性改动太大导致项目不可用。

最后在所有资源都迁移完成后，移除 Resources 文件夹中的资源，彻底切换到 YooAsset。这时你会发现项目的资源管理变得清晰和可控，为将来的热更新和优化打好了基础。

## 重构路线图

### 第一阶段：搭建基础架构

重构的第一步是搭建新架构的基础框架。将架构代码整合到项目中，包括 IArchitecture 接口、Architecture 基类、Model、System、Controller、Command、Query 等核心类。创建你的游戏专用的 Architecture 类，比如 GameArchitecture，在其中注册初始的 Model、System、Utility。

在这个阶段还要建立架构的初始化流程。在游戏启动时初始化架构，确保所有的 Model、System、Utility 都被正确注册。你可以创建一个 ArchitectureInitializer 脚本来处理这个过程。

这个阶段不要急于重构现有功能，而是让新旧架构能够共存。现有的单例管理器继续工作，新架构作为补充存在。这样可以保证项目在重构过程中始终是可运行的。

### 第二阶段：迁移独立模块

选择一些相对独立的模块开始迁移。音效系统是一个好的起点，它比较简单，依赖关系少。将 AudioManager 改造成 AudioSystem，让它通过架构注册而不是使用单例。修改调用 AudioManager 的地方改为通过架构获取 AudioSystem。

配置管理系统也可以在这个阶段迁移。将你的 JsonManager 改造成 JsonUtility，创建各种配置读取的 Utility，比如 ItemConfigUtility、QuestConfigUtility 等。这些 Utility 在架构初始化时注册，其他模块通过 GetUtility 来获取。

特效系统的迁移类似于音效系统。这些相对独立的模块迁移完成后，你会对新架构的使用有了初步的体会，为后续迁移更复杂的模块打下基础。

### 第三阶段：重构核心数据模型

在这个阶段开始创建核心的 Model 类。PlayerModel 存储玩家的核心数据，InventoryModel 管理背包数据，QuestModel 管理任务数据等。这些 Model 在创建初期可以与原有的管理器并存，逐步将数据迁移过来。

创建 Model 时要仔细设计数据结构，决定哪些数据应该用 BindableProperty 包装，哪些方法应该提供给外部使用。Model 的接口设计要考虑到未来的扩展性，但也不要过度设计，从实际需求出发。

在 Model 创建完成后，让一些新的功能开始使用这些 Model。比如创建一个新的 UI 面板，让它直接从 Model 读取数据并监听变化。这样可以验证 Model 的设计是否合理，同时让团队成员熟悉新的开发方式。

### 第四阶段：重构业务逻辑层

这个阶段开始创建 System 类来处理业务逻辑。将原来分散在各个管理器中的业务逻辑提取出来，按照新的设计原则重新组织。

比如将任务相关的业务逻辑提取到 QuestSystem 中，包括任务的接取检查、奖励发放、进度更新等。CombatSystem 处理战斗相关的逻辑，包括伤害计算、状态效果等。

在创建 System 时要注意职责的划分，避免创建上帝类。如果一个 System 变得很大，可能说明它承担了太多职责，需要考虑拆分。同时要注意 System 之间的依赖关系，避免循环依赖。

### 第五阶段：重构表现层

表现层的重构涉及到 MonoBehaviour 组件的改造。将原来直接访问管理器单例的代码改为通过 IController 接口访问架构。

UI 面板改造成 Controller，让它们能够获取 Model、发送命令、监听事件。这个过程可能需要一些调整，因为原来的 UI 代码可能直接调用管理器的方法，现在需要改成发送命令。

角色的各个功能组件继续保持组件化的设计，但它们通过父组件的 Controller 接口来访问架构。输入处理改造成 InputController，监听输入事件并转化为命令。

### 第六阶段：处理动作策略系统

这个阶段专门处理你的基于 ScriptableObject 的动作策略系统。按照我们讨论的方案，创建最小化的上下文对象，让 CharacterManager 从 Model 同步必要的状态。

修改动作策略的基类，让它接收上下文对象而不是直接接收 CharacterManager。调整具体的动作策略类，让它们能够从上下文获取需要的信息。

创建 PerformWeaponActionCommand 来封装武器动作的执行流程，在命令中处理业务逻辑检查和状态修改，然后调用动作策略来执行表现逻辑。

重构输入系统，让 PlayerInputController 发送命令而不是直接调用武器的动作方法。这个过程需要仔细测试，确保所有的动作都能正常工作，连击系统、冲刺攻击等特殊情况都要验证。

### 第七阶段：整合资源管理

在这个阶段引入 YooAsset 作为统一的资源管理方案。首先配置 YooAsset，创建资源包配置，将资源按功能分组。

创建 ResourceSystem 来封装 YooAsset 的使用，提供统一的资源加载接口。然后逐个模块迁移资源加载代码，从 Resources 或 AssetBundle 改为使用 ResourceSystem。

对象池系统在这个阶段与资源管理系统整合。PoolSystem 使用 ResourceSystem 来加载预制体，创建对象池。配置管理的 Utility 也改为使用 ResourceSystem 加载 ScriptableObject。

最后清理 Resources 文件夹，移除不再需要的 AssetBundle 加载代码，完全切换到 YooAsset。

### 第八阶段：完善和优化

在这个阶段重点是完善各个系统，修复在重构过程中发现的问题。补充缺失的功能，优化性能瓶颈。

完善存档系统，确保所有需要持久化的数据都能正确保存和加载。测试多存档功能，验证不同存档之间的数据隔离。

完善事件系统，梳理所有的事件类型，确保事件的命名清晰，事件携带的信息完整。检查是否有遗漏的监听注销，避免内存泄漏。

优化资源加载，分析资源的使用情况，合理配置资源分组和压缩方式。测试资源的加载性能，优化加载流程。

编写文档，记录架构的使用方式，各个模块的职责，常见问题的解决方案等。这些文档会帮助团队成员更快地适应新架构。

### 第九阶段：移除旧代码

当所有功能都迁移到新架构后，可以开始移除旧的代码了。删除不再使用的单例管理器，删除旧的事件系统，删除冗余的外观类等。

这个过程要谨慎，确保删除的代码确实不再被使用。可以先注释掉而不是直接删除，运行一段时间确认没有问题后再彻底删除。

移除旧代码后项目会变得更加清爽，代码量减少，结构更加清晰。这时整个重构过程就完成了，你会拥有一个基于清晰架构的项目，为后续的开发和维护打下了坚实的基础。

## 最佳实践和注意事项

### 渐进式重构的原则

重构最重要的原则是保持项目始终可运行。不要试图一次性改动所有代码，那样风险太大。应该每次只重构一个模块，改完测试通过后再进行下一个。

在重构过程中可以让新旧代码并存。比如 AudioManager 单例和 AudioSystem 可以同时存在一段时间，旧代码继续使用单例，新代码使用 System。等所有调用都迁移完后再删除单例。

每完成一个阶段就提交代码到版本控制系统。这样如果后续发现问题，可以方便地回退到之前的状态。好的版本控制习惯在重构时特别重要。

不要在重构的同时添加新功能。重构和功能开发应该分开进行，这样可以保持专注，也便于验证重构没有引入问题。

### 测试策略

在重构过程中要持续测试。每改完一个模块就运行游戏，验证这个模块的功能是否正常。不要等所有改完了再测试，那样一旦出问题很难定位。

可以编写一些自动化测试来验证核心功能。比如测试 InventoryModel 的添加物品、移除物品功能，测试 QuestModel 的任务进度更新功能等。这些测试可以在重构后快速验证功能的正确性。

多人协作时要特别注意测试。不同人可能同时在重构不同的模块，要确保改动不会相互影响。定期集成代码并运行完整的测试可以及早发现问题。

### 团队协作

如果是团队开发，要做好沟通和分工。可以一个人负责搭建架构框架，其他人在框架就绪后各自负责不同模块的重构。定期开会同步进度，讨论遇到的问题。

建立编码规范，约定命名方式、文件组织、注释风格等。统一的代码风格让团队成员更容易理解彼此的代码。

及时更新文档，将重构过程中的决策、解决方案、经验教训记录下来。这些文档不仅帮助当前的重构，也为将来的维护和新成员的加入提供了宝贵的资料。

### 性能考虑

新架构引入了一些间接层，比如通过架构获取 Model 和 System，使用事件系统通信等。这些会带来一些性能开销，但在大多数情况下这个开销是可以接受的。

对于性能关键的代码，比如每帧都在执行的逻辑，可以缓存获取的 Model 和 System 引用，避免每次都通过架构查找。对于频繁触发的事件，考虑使用直接调用而不是事件通知。

使用性能分析工具来检测瓶颈。Unity 的 Profiler 可以帮助你找到性能问题所在。如果发现某个地方是瓶颈，有针对性地优化，而不是过早优化。

记住架构的主要目的是提高代码的可维护性和可扩展性，而不是极致的性能。在大多数情况下，清晰的架构带来的开发效率提升远远超过一点性能开销。只有在确实出现性能问题的地方才需要权衡取舍。

### 常见陷阱

避免过度设计。不要为了使用架构而使用架构，不要把简单的事情复杂化。如果一个功能用简单的方式就能实现，就不要强行套用复杂的模式。

避免上帝类。不要创建一个巨大的 Model 或 System 包含所有功能。如果一个类变得很大，说明它承担了太多职责，需要考虑拆分。

避免循环依赖。如果 SystemA 需要 SystemB，SystemB 又需要 SystemA，说明职责划分有问题。重新思考设计，可能需要提取一个第三方的 System 或者使用事件来解耦。

注意内存泄漏。事件监听、BindableProperty 监听都需要在合适的时候注销。使用框架提供的 UnRegisterWhenGameObjectDestroyed 可以自动处理大部分情况，但对于非 GameObject 的对象要特别注意手动注销。

保持架构的一致性。一旦决定了某种设计方式，就在整个项目中保持一致。不要在有的地方用命令，有的地方直接调用方法，有的地方用事件。一致的风格让代码更容易理解和维护。

## 总结

这份重构指南描述了如何将你的 ARPG Demo 项目从传统的单例管理器架构迁移到基于分层架构的设计。这是一个系统的工程，需要时间和耐心，但最终会带来更加清晰、可维护、可扩展的代码库。

记住重构的核心原则是渐进式和持续测试。不要急于求成，而是稳扎稳打，每一步都确保项目可以正常运行。保留你原有设计中的优秀部分，比如基于 ScriptableObject 的策略模式和数据配置系统，让它们与新架构和谐共存。

新架构不是银弹，它不会解决所有问题。但它提供了一个清晰的思维框架来组织代码，让你在面对复杂性时有章可循。随着项目的发展，你会越来越体会到清晰架构带来的价值。

祝你重构顺利，如果在重构过程中遇到具体的问题，可以随时参考这份指南，或者寻求进一步的帮助。这个旅程可能有挑战，但最终的结果会让这一切努力都值得。
